## 1. Durability Qdrant — защита индекса от порчи

**Проблема.** Поймали вживую (2026-06-12): storage Qdrant (RocksDB) повредился
(`IO error: While open a file for random read`), поиск отдавал 500. Причина — контейнер
Qdrant убили жёстко (SIGKILL / OOM / жёсткий рестарт) во время записи, RocksDB остался в
неконсистентном состоянии. Данные не потеряли (**Postgres — source-of-truth**), но был
простой + ручная пересборка `reindex --recreate`. Сейчас защиты нет: не гарантирован
graceful shutdown, нет лимитов памяти от OOM-kill, нет снапшотов Qdrant.

**Почему отложено.** Текущий прод на VibeNest — временный; серьёзную надёжность добиваем
на серверах заказчика.

**Варианты решения (на серверах заказчика):**
- **Graceful shutdown** — Qdrant получает SIGTERM + достаточный grace-period до SIGKILL
  (`stop_grace_period` в compose / terminationGracePeriod в оркестраторе). По SIGTERM Qdrant флашит RocksDB.
- **Лимиты памяти с запасом** — память контейнера выше пикового рабочего набора, чтобы
  OOM-killer не прибивал Qdrant в момент записи. Мониторить RSS.
- **Снапшоты Qdrant** — нативный snapshot API (`POST /collections/{name}/snapshots`) по
  расписанию в надёжное хранилище. Restore быстрый — дешевле полной переиндексации.
- **Авто-восстановление на старте** — детект порчи при старте → авто `reindex --recreate`
  из PG (механизм уже есть, остаётся повесить на health-gated хук).
- **Floor уже есть:** PG (SoT) + `reindex --recreate`. Durability лишь снижает частоту порчи и время простоя.

## 2. Косметика по запросу смежников

Три мелочи — делаем, если Группа 3 / оценочная группа попросят.

### 2a. `published_ts` в ответе поиска
**Проблема.** Дата публикации лежит в payload Qdrant и в PG (есть даже recency-boost по дате),
но в ответе `/search` (`Hit`) её **не возвращаем**. Группа 3 не может показать «актуально на
дату», отсортировать/отфильтровать по свежести у себя, процитировать давность источника —
пришлось бы делать второй запрос.

**Варианты:**
- **Минимально (рекоменд., ~30 мин):** добавить `published_ts` в `SearchHit` + маппинг ответа
  REST. Данные уже подняты из payload — это пара строк, без новых запросов.
- **Шире:** вернуть весь блок метаданных родителя, если понадобится ещё что-то.
- **Никак:** если дата в ответе не нужна — не трогаем.

### 2b. Warm-up первого запроса
**Проблема.** Модель грузится на старте (в памяти ещё до открытия порта), но **первый**
embed-вызов делает ленивую инициализацию (прогрев вычислений, пул потоков) — первый запрос
после рестарта чуть медленнее последующих. С `st-bm25` это уже немного (и dense, и BM25
грузятся на старте — не как раньше ленивый ~7 с), но на демо / сразу после редеплоя первый
пользователь ловит лаг.

**Варианты:**
- **Прогрев на старте (рекоменд., ~10 строк):** один холостой `embed_query("warmup")`
  (+ опц. dummy-search) до открытия порта → первый реальный запрос уже тёплый.
- **Readiness-gate:** не отдавать `/readyz` ok, пока прогрев не прошёл → платформа не шлёт
  трафик в холодный pod.
- **Никак:** принять разовый лаг (с st-bm25 он малый).

### 2c. Счётчик точек Qdrant в `/stats`
**Проблема.** После reindex (эпизод с 504) не было способа в один запрос подтвердить, что в
Qdrant ровно 1394 точки — `/stats` берёт счётчики из **Postgres** (SoT), а не из реального
индекса Qdrant. Полноту пришлось выводить косвенно (поиск работает → значит ок). После
reindex/порчи хочется быстрый чек «PG: 1394 чанка, Qdrant: 1394 точки → в синхроне».

**Варианты:**
- **Добавить кол-во точек Qdrant в `/api/v1/stats` (рекоменд., ~15 мин):** `client.count(collection)`
  → поле `qdrant_points` рядом с `total_chunks`. Рассинхрон (PG ≠ Qdrant) виден мгновенно.
- **По источникам:** count с фильтром по `source_id` — проверять полноту конкретного источника.
- **В health:** опц. предупреждение в `/readyz` при дрейфе PG ≫ Qdrant.

## 3. Semantic-чанкинг

**Проблема.** В сравнении методов нарезки (Задача 1) был `semantic`-сплиттер (LlamaIndex
`SemanticSplitterNodeParser` — режет по смысловой близости соседних предложений, а не по
фикс-размеру). Остался **заглушкой**: не прогнан, нужен `llama-index` (pip/сеть). В лидерборде
нет его цифр.

**Необходимость — почти нулевая.** `recursive` уверенно победил (R@5 0.98), он у потолка.
Semantic дороже (эмбедит каждое предложение ради границ) и вряд ли обгонит.

**Варианты:**
- **Оставить как есть (рекоменд.):** recursive победил — не тратим время.
- **Если интересно (только ресёрч):** поставить `llama-index` в bench-venv, снять заглушку
  `SemanticChunker.split()`, прогнать, добавить строку в лидерборд. Это в `leaderboard/`, не в прод-сервисе.
- **Удалить заглушку:** если точно не вернёмся — убрать, чтоб не путала.
