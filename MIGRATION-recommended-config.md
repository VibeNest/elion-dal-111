# Миграция на рекомендуемую конфигурацию RAG

Внедрение результатов исследования (см. `leaderboard/FINAL_CONFIG.md`) в прод elion_dal.

## Что изменилось (дефолты)

| Параметр | Было | Стало |
|---|---|---|
| `EMBEDDING_BACKEND` | `fastembed` (e5-large + BM25) | **`st-bm25`** (USER-bge-m3 + BM25) |
| `EMBEDDING_MODEL` | пусто | **`deepvk/USER-bge-m3`** |
| `CHUNK_TOKENS` / `CHUNK_OVERLAP` | 400 / 64 | **1024 / 102** |
| Реранкер | опция (off) | **удалён из кода** |

Обоснование: бенч показал hybrid **USER-bge-m3 + BM25 → Recall@5 0.98** (лучший конфиг);
реранкер на CPU не окупается (R@5 падает, ~9 c/запрос), а GPU у заказчика нет. Метод поиска
(hybrid dense+sparse, RRF) в проде уже был — менялись только модель, размер чанка и чистка реранкера.

Новый бэкенд `st-bm25`: dense = `deepvk/USER-bge-m3` (sentence-transformers), sparse = BM25
(`Qdrant/bm25`, fastembed); слияние RRF на стороне Qdrant. Файл: `embedding/st_bm25_provider.py`.

## Шаги миграции (однократно)

1. **Зависимости.** Ядро теперь требует `sentence-transformers` (тянет torch):
   ```
   pip install -e .
   ```
2. **Модели.**
   - `deepvk/USER-bge-m3` — уже в HF-кэше (`HF_HOME=D:\hf_home`).
   - `Qdrant/bm25` (sparse, fastembed) — **скачается при первом запуске** (~несколько МБ). Это
     единственная сетевая операция; при включённом троттлинге допустима.
3. **Поднять инфраструктуру:** Qdrant + Postgres.
4. **Переиндексация из источника (ОБЯЗАТЕЛЬНО).** Сменились И модель, И размер чанка, поэтому нужна
   полная переиндексация с **перенарезкой**:
   - Пересоздать коллекцию и заново загрузить документы из источника (платформенный sync / ETL /
     `UpsertDocuments`) — это пере-чанкит на 1024/102 и переэмбеддит новым провайдером.
   - ⚠️ `python -m elion_dal.cli.reindex --recreate` НЕ подходит как единственный шаг: он
     переэмбеддивает СУЩЕСТВУЮЩИЕ чанки из PG (старый размер 400), т.е. применит новую модель, но
     НЕ новый размер чанка. Нужен повторный ingest из источника.
5. **Проверка:**
   - `HealthCheck` → `embedding_backend == "st-bm25"`.
   - `Search` по типовому запросу возвращает релевантные hits, `dense_score > 0`.

Примечание: размерность dense осталась 1024 (USER-bge-m3 = e5-large), а sparse — BM25 с IDF, как и у
прежнего fastembed-дефолта, поэтому схема коллекции совместима. Пересоздание (`--recreate` / re-ingest)
всё равно нужно, т.к. старые векторы посчитаны другой моделью.

## Откат

Вернуть в `.env` (или в админке, restart-настройки) и переиндексировать из источника:
```
EMBEDDING_BACKEND=fastembed
EMBEDDING_MODEL=
CHUNK_TOKENS=400
CHUNK_OVERLAP=64
```
